# 阶段 A / B 与 Swagger 清理设计文档

## 目标

在不触碰 RBAC 后续接口的前提下，完成 `pending-tasks` 中阶段 A（功能补全与体验修复）、阶段 B（工程化基础）的首轮落地，并彻底移除 Swagger 相关入口与静态文档暴露。

## 本轮范围

本轮只覆盖以下内容：

- P-20：补回博客详情页“上一篇 / 下一篇”导航
- P-22：排查并修复前台滚动黑线问题
- P-25：确认并微调管理后台视觉一致性
- P-02：环境配置拆分
- P-01：Docker 化
- P-03：测试补齐
- Swagger 清理：移除页面入口、静态文件暴露、遗留 handler 和不再需要的文档产物引用

本轮明确不做：

- P-06 ~ P-12 RBAC 管理接口
- Kitex 拆分
- MinIO / Redis 接入
- About 页后台可编辑

## 当前项目背景

当前站点已经完成纯静态 SPA 迁移，前台和后台都通过 `static/` 目录交付，后端只承担 API 和静态资源分发。

当前遗留问题主要分成两类：

1. 用户可感知问题：详情页缺失上下篇导航、前台滚动黑线、后台视觉仍需对齐。
2. 工程化缺口：配置模型还比较粗、缺少 Docker 交付、测试几乎为空。

Swagger 目前仅剩三部分遗留：

- `main.go` 中对 `/swagger.json` 与 `/swagger.yaml` 的暴露
- `biz/site/handler.go` 中保留的 Swagger HTML
- `docs/` 目录中的 swagger 文档文件

## 备选方案

### 方案 1：先做阶段 A + 去 Swagger，再做阶段 B

先完成用户可见问题与 Swagger 清理，确认站点行为稳定后，再补工程化基础。

优点：

- 先收掉最直接影响体验的遗留项
- Swagger 删除和路由清理改动集中、易验证
- 阶段 B 可以基于稳定功能面再补测试和交付

缺点：

- 会分成两个实现批次

### 方案 2：A、B 与 Swagger 一次性打包完成

优点：

- 只经历一次集中修改

缺点：

- 配置、部署、前端、后端同时变化，回归风险高
- 一旦出问题，定位成本更高

### 方案 3：先做阶段 B，再回头处理阶段 A

优点：

- 工程化地基先补齐

缺点：

- 用户可见遗留项继续保留
- 测试会先围绕一个已知不完整的行为面建立

## 采用方案

采用方案 1。

执行顺序固定为：

1. 先完成 P-20、P-22、P-25 与 Swagger 清理
2. 再完成 P-02、P-01、P-03

这样可以保证“功能面先稳定，再固化工程化交付方式”。

## 设计说明

### 1. 上下篇导航

详情页的上一篇 / 下一篇不再维持前端假数据占位，而是改为真实接口驱动。

后端在文章 service 中新增“基于当前已发布文章按时间顺序查相邻文章”的查询能力，返回最小展示模型：`id`、`title`、`slug`、`published_at`。前端 `BlogAPI.getAdjacentPosts` 调用该接口，详情页只在拿到相邻文章时渲染链接。

排序规则以当前公开文章列表的主排序保持一致：优先 `is_top DESC`，再按 `published_at DESC, created_at DESC`。这样详情页跳转关系与列表阅读顺序一致，避免认知错位。

### 2. 滚动黑线修复

黑线问题优先按前台样式层处理，不通过大改布局结构来冒险。先排查 `body`、主容器、背景层、TOC/浮动按钮等使用 `transform`、`overflow`、固定定位和视口宽度的区域，重点去掉容易触发合成缝隙的组合。

修复策略遵循“最小 CSS 变更”：

- 统一背景承载层
- 收紧横向溢出
- 避免对整页主容器施加不必要的 `transform`
- 对浮层单独处理而不是带动页面整体合成

验证以真实页面滚动表现为准，不做纯静态猜测。

### 3. 后台视觉对齐

后台不做重新设计，只做和现有 Terminal Gallery 风格的收口对齐。目标是保证：

- 登录、列表、编辑器、分类标签面板的视觉语言一致
- 关键按钮、状态标签、空状态、加载态不突兀
- 不引入新的交互模型

实现上只允许局部 CSS 和必要的文案/结构微调，避免把这轮工作扩成一次新的后台重构。

### 4. Swagger 清理

Swagger 视为已废弃能力，整轮直接清空：

- 删除 `/swagger.json`、`/swagger.yaml` 暴露
- 删除 `biz/site/handler.go` 中 Swagger UI
- 清理 `biz/site/route.go` 中相关入口
- 删除或停用 `docs/swagger.json`、`docs/swagger.yaml`、`docs/docs.go`

目标状态是：仓库和运行中的站点都不再暴露 Swagger 页面与文档文件。

### 5. 配置拆分

配置拆分不追求引入复杂配置框架，继续沿用当前 `yaml + env override` 模式，但把配置结构从“只有 server/mysql/session”扩成更明确的领域分组，至少覆盖：

- server
- mysql
- session
- upload
- site

同时保留环境变量覆盖，以兼容当前本地开发习惯和 Docker 注入方式。

### 6. Docker 化

Docker 目标是最小可运行交付，不做多容器编排扩张。交付内容包括：

- 应用镜像构建
- 基础运行说明
- 可选的 `docker-compose.yml` 用于本地联动 MySQL

镜像应直接服务当前静态前端和 Go API，不引入额外反向代理。

### 7. 测试策略

测试本轮聚焦“关键路径回归”，不追求全覆盖。优先补：

- 文章相邻导航 service 逻辑
- 配置加载与 env override
- 站点关键 handler / route 的基础行为

如前端可低成本抽离纯函数，也可补最小 JS 单测；否则以后端测试为主。

## 错误处理

- 新增相邻文章接口延续现有 `errno` 与 `response` 统一响应模式
- Swagger 删除后，访问旧地址应返回正常的站点 404/非暴露资源行为
- 配置缺省值仍应保证本地可启动，不因拆分直接破坏开发体验

## 验证方式

- 先写失败测试，再补实现
- Go 侧以 `go test ./...` 为基础验证
- 如引入 Docker 文件，至少验证构建命令和配置说明自洽
- 对黑线修复和后台视觉对齐，保留人工页面验证

## 完成标准

- 博客详情页可展示上一篇/下一篇导航
- 前台滚动黑线问题消失或显著收敛
- 后台视觉完成一轮收口对齐
- Swagger 页面和文档不再可访问
- 配置结构完成拆分且兼容 env override
- 项目可通过 Docker 启动
- 仓库新增最关键回归测试，`go test ./...` 可通过
