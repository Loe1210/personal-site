# 前端抽离与 Nginx 入口设计

## 目标

把当前静态前端从后端仓库中独立出来，改成单独的 Docker 服务，由 Nginx 提供静态资源访问，并统一转发 `/api/*` 到网关服务。

这样做的结果是：

- 浏览器只需要访问一个入口
- 前端部署和后端部署解耦
- 后端继续保留网关、BFF、业务服务的职责
- 以后前端可以独立扩容、独立替换、独立发版

## 现状

当前项目里已经有一套静态页面资源，目录主要在 `static/` 下，同时后端已经有：

- `gateway` 作为统一 API 网关
- `web-bff` 作为 Web 聚合层
- `auth-service`、`content-service`、`media-service` 作为业务服务

现在的问题是：

- 浏览器没有一个明确的前端入口
- 静态资源仍然散落在后端仓库中
- 前端访问路径和后端 API 入口没有被清晰分层

## 方案

### 1. 新增前端 Docker 服务

新增一个独立的前端服务，比如命名为 `frontend`。

该服务只负责：

- 提供静态文件
- 作为浏览器唯一入口
- 把 `/api/*` 请求转发给 `gateway`

### 2. Nginx 作为前端入口

前端容器内使用 Nginx 承担两件事：

- 直接返回静态页面
- 代理后端接口请求

推荐规则：

- `/` 和所有静态资源 -> 前端静态文件
- `/api/*` -> `gateway:8888`

### 3. 后端职责不变

后端仍然按现有微服务职责划分：

- `gateway`：统一 API 入口
- `web-bff`：面向前端页面的数据聚合
- `content-service`：文章、分类、标签等内容能力
- `auth-service`：登录鉴权
- `media-service`：图片、上传、媒体能力

前端不直接访问任何业务服务，只访问网关。

## 推荐访问方式

浏览器访问：

`http://127.0.0.1:8080`

其中：

- `8080` 对外暴露给浏览器
- Nginx 在容器内监听 `80`
- 容器内部转发到 `gateway:8888`

## 目录建议

前端相关文件建议单独整理为：

```text
frontend/
├── nginx/
│   └── default.conf
├── static/
│   ├── index.html
│   ├── blog/
│   ├── admin/
│   └── assets/
└── Dockerfile
```

如果暂时不想移动太多文件，也可以先继续复用仓库里的 `static/`，只是在构建前端镜像时把它们复制进去。

## Docker Compose 调整

建议在 `deploy/docker/compose.yaml` 中新增前端服务：

- `frontend`
- 端口映射 `8080:80`
- 依赖 `gateway`

这样本地启动后：

- 前端通过 `http://127.0.0.1:8080`
- 后端 API 通过 `http://127.0.0.1:8080/api/...`

## 需要保留的能力

前端抽离后，仍然要保留：

- 登录态依赖的 cookie/session 机制
- `/api` 路由下的统一鉴权
- 页面访问后调用文章、分类、标签等接口

## 风险与约束

- Nginx 配置必须保证前端路由刷新不 404
- 静态资源路径不能和 `/api` 冲突
- 如果前端页面使用绝对路径，要统一检查 base path
- 容器内网地址要保持稳定，避免前端镜像写死宿主机地址

## 验收标准

满足以下条件即可认为这个改造完成：

1. 浏览器访问 `http://127.0.0.1:8080` 能打开前端页面
2. 页面上的接口请求会自动走 `/api`
3. `/api` 请求能被 Nginx 正确转发到网关
4. 原有后端服务仍可通过 Docker Compose 正常启动
5. 页面刷新和直接访问子路由不会报 404
