# Phase 03

## 目标

将文章模块从内存数据版本迁移到 MySQL 持久化版本，让文章后台 CRUD 和公开读取都基于真实数据库运行。

## 完成内容

1. 选定 `MySQL + GORM` 作为当前单体项目的数据持久化方案
2. 增加数据库配置结构与环境变量加载逻辑
3. 新增 `biz/dal/db` 数据库初始化代码，并在程序启动时完成连接建立
4. 接入 `gorm.io/gorm` 与 `gorm.io/driver/mysql` 依赖
5. 为文章模块建立数据库模型 `biz/dal/db.Article`
6. 启动时通过 `AutoMigrate` 自动创建 `articles` 表
7. 将公开文章列表接口 `GET /api/articles` 切换为数据库查询
8. 将公开文章详情接口 `GET /api/articles/:slug` 切换为数据库查询
9. 将后台文章列表接口 `GET /api/admin/articles` 切换为数据库查询
10. 将后台文章创建接口 `POST /api/admin/articles` 切换为数据库写入
11. 将后台文章更新接口 `PUT /api/admin/articles/:id` 切换为数据库更新
12. 将后台文章删除接口 `DELETE /api/admin/articles/:id` 切换为数据库删除
13. 移除文章模块对内存假数据 `mockArticles` 的依赖，完成文章模块持久化迁移

## 设计决策

### 1. 当前阶段优先采用 MySQL

为了更贴近后续业务开发和实习场景，本阶段选择 `MySQL` 而不是 `SQLite`。这样可以更早适应真实服务开发中的连接管理、建表和持久化读写方式。

### 2. 先使用 GORM 跑通持久化闭环

当前阶段的重点是把单体项目真正从“接口练习版”推进到“可持续迭代版”，因此优先使用 `GORM` 快速打通：

- 建模
- 自动迁移
- 基础 CRUD

后续如果需要更细的 SQL 控制，再补 repository 或 query 层抽象。

### 3. tag_ids 暂时存为字符串

为了先完成文章主链路的持久化迁移，`tag_ids` 当前阶段先存为逗号拼接字符串，而不是立即拆成关联表。这样可以避免过早引入标签关系设计，让本阶段聚焦在文章数据本身。

### 4. 环境变量先满足开发可用

数据库配置目前通过环境变量读取，先保证开发链路和本地运行体验稳定。后续如果项目需要更完整的配置管理，再引入配置文件或更系统的加载方式。

## 遇到的问题

### 1. Windows 用户环境变量写入后当前终端没有立即生效

虽然用户环境变量已经成功写入注册表，但当前 PowerShell 进程未立即拿到最新值，导致程序第一次启动时没有读到 `MYSQL_PASSWORD`。临时通过当前终端补变量继续开发，后续再通过重新打开终端或重启系统解决刷新问题。

### 2. MySQL 连接错误需要区分“密码错误”和“未读取到密码”

报错 `using password: NO` 并不表示密码填错，而是 DSN 中根本没有带密码。这个问题通过确认环境变量读取状态后定位清楚。

### 3. 内存版和数据库版逻辑并存时容易产生双源数据混淆

在切换过程中，必须明确哪些接口已经走数据库，哪些还依赖内存数据。最终通过把后台 CRUD 也切到 MySQL，并移除 `mockArticles`，消除了双源数据问题。

## 当前结果

当前项目已经具备以下能力：

- Hertz 单体服务可正常连接 MySQL 启动
- `articles` 表可自动建表
- 公开文章列表和详情接口走真实数据库
- 后台文章 CRUD 全部走真实数据库
- Swagger 可继续作为统一调试入口验证数据库版接口
- 文章模块已经完成从内存实现到持久化实现的迁移
