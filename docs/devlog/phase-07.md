# Phase 07

## 目标

完成上传模块第一版落地，建立本地文件存储版图片上传能力，并让上传结果与文章封面字段形成闭环。

## 完成内容

1. 新增 `UploadFile` 数据库模型并接入 `AutoMigrate`
2. 用 `upload.thrift` 生成上传模块接口模型
3. 新增上传 service，完成上传记录查询能力
4. 新增图片上传主流程：校验、生成文件名、落盘、写库、返回 URL
5. 新增上传 handler，完成表单字段绑定与文件获取
6. 新增上传 router，挂载后台上传和上传记录查询接口
7. 在 `main.go` 中开放 `/static` 静态资源访问
8. 完成 Swagger 联调，验证上传、查询和静态访问链路
9. 打通上传结果与文章 `cover_image` 字段联动
10. 明确上传模块当前只做稳定版小文件上传
11. 将大文件相关能力记录为后续优化项

## 设计决策

### 1. 上传与文章解耦

当前设计中：

- 上传模块负责保存文件并返回 `file_url`
- 文章模块只保存 `cover_image`

这样做可以保持边界清晰，后续如果切对象存储或独立上传服务，文章侧不需要改业务模型。

### 2. 当前阶段只做稳定版小文件上传

当前项目目标是个人小站，不是生产级大文件平台。因此本阶段只实现：

- 图片上传
- 大小限制
- MIME 白名单校验
- 流式落盘
- 元数据入库

而不在第一版中引入分片上传、断点续传、状态补偿等复杂机制。

### 3. 上传文件不进仓库

上传目录属于运行时产物，因此将 `static/uploads/` 排除出 Git，避免测试图片和未来运行时文件进入版本库。

## 遇到的问题

### 1. 数据库模型与 thrift 生成模型字段命名不完全一致

例如：

- 数据库模型使用 `ID`、`FileURL`
- thrift 生成模型使用 `FileID`、`FileURL`

最终通过 service 层显式映射解决，而不是强行让两套模型完全一致。

### 2. 上传能力的实现边界需要提前定清楚

在开发过程中确认：第一版不做大文件体系，而是先保证小文件上传稳定可用。这避免了当前阶段过早引入复杂设计。

## 当前结果

当前项目已经具备以下上传相关能力：

- 后台图片上传
- 上传记录查询
- 静态资源访问
- 上传结果回填文章封面图
- 上传异常场景基础拦截
