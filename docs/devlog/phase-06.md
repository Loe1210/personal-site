# Phase 06

## 目标

完成文章内容域规范化，把文章与标签关系从过渡期的字符串字段方案，收口为正式的关系表方案，并保证文章写操作具备事务一致性。

## 完成内容

1. 删除 `Article` 模型中的 `TagIds string` 过渡字段
2. 保留并正式启用 `ArticleTag` 关系表作为文章标签唯一数据来源
3. 在文章 service 中新增标签去重辅助逻辑
4. 在文章创建前增加标签存在性校验
5. 在文章更新前增加标签存在性校验
6. 在文章创建和更新时通过 `article_tags` 维护标签关系
7. 在文章详情与列表返回时，从 `article_tags` 回填 `tag_ids`
8. 为文章创建、更新、删除三类写操作补充事务保护
9. 将文章删除逻辑扩展为同步清理 `article_tags`
10. 手动清理数据库中历史遗留的 `articles.tag_ids` 字段
11. 完成文章创建、更新、删除和异常场景的接口验证

## 设计决策

### 1. 放弃字符串存 tag，改为关系表存 tag

旧方案把 `tag_ids` 以逗号字符串形式存在 `articles` 表里，虽然短期可用，但会带来：

- 标签关系不规范
- 后续扩展标签筛选和联表查询困难
- 前后台数据源不一致

因此本阶段正式改为：

- `articles` 只保存文章自身字段
- `article_tags` 负责保存文章与标签的关联关系

### 2. 标签关系更新采用“先删后插”策略

在当前单体阶段，为了降低实现复杂度，文章标签关系更新采用：

- 先删除旧关系
- 再插入新关系

这种方式简单、稳定，也足够支撑个人站当前规模。

### 3. 写操作统一补事务

文章写操作天然包含“主表 + 关系表”两部分。如果不加事务，容易出现半成功状态。因此本阶段将：

- `CreateArticle`
- `UpdateArticle`
- `DeleteArticle`

统一纳入事务保护，保证数据一致性。

## 遇到的问题

### 1. GORM AutoMigrate 不会主动删除旧列

即使已经从 Go 结构体里删除了 `TagIds` 字段，数据库中的 `articles.tag_ids` 历史列仍然会保留。最终通过手动执行：

```sql
ALTER TABLE articles DROP COLUMN tag_ids;
```

完成了旧列清理。

### 2. 标签存在性校验需要先去重

如果请求中的 `tag_ids` 包含重复值，直接按长度比对数据库查询数量会误判标签不存在。因此在校验前增加了去重逻辑。

## 当前结果

当前文章域已经具备以下能力：

- 文章标签关系通过 `article_tags` 维护
- 文章读接口返回的 `tag_ids` 来自关系表而不是冗余字段
- 文章创建、更新、删除具备事务一致性
- 标签不存在、slug 冲突等异常场景能够按预期返回
