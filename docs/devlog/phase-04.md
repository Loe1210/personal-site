# Phase 04

## 目标

为文章模块补齐分类与标签的基础关系能力，让文章不再只依赖简化字段，而是逐步建立真实的内容组织结构。

## 完成内容

1. 新增 `category.thrift` 并生成分类领域 model
2. 新增 `tags.thrift` 并生成标签领域 model
3. 建立 `categories` 表并接入自动迁移
4. 建立 `tags` 表与 `article_tags` 关联表并接入自动迁移
5. 实现并验证 `POST /api/admin/categories`
6. 实现并验证 `GET /api/categories`
7. 实现并验证 `GET /api/admin/categories`
8. 实现并验证 `POST /api/admin/tags`
9. 实现并验证 `GET /api/tags`
10. 实现并验证 `GET /api/admin/tags`
11. 为文章创建与更新增加 `category_id` 存在性校验
12. 将旧的 `proto` 文件和过渡期 `pb.go.bak` 文件从仓库中移除，统一到当前 `thrift` 路线

## 设计决策

### 1. 先做 category，再做 tag

分类是最简单的一对多关系，标签是更复杂的多对多关系。先把 `category` 跑通，再引入 `tag` 和关联表，可以把复杂度拆开，避免一次改动过大。

### 2. 先做标签模块自身闭环，再接文章多对多关系

当前阶段先完成：

- 标签建模
- 标签表迁移
- 标签创建与列表接口

这样可以先确认标签模块本身稳定，再在下一阶段把文章 `tag_ids` 从字符串方案切换到 `article_tags` 关联表。

### 3. category_id 必须引用真实分类

文章创建和更新不再接受“任意数字”作为分类 ID，而是要求分类必须真实存在。这样文章和分类的关系才真正成立，也为后续前台展示和后台管理打下基础。

## 遇到的问题

### 1. 新建 category.thrift 前直接执行生成命令会失败

`hz model` 依赖真实存在的 IDL 文件，因此必须先创建 `idl/category.thrift`，再执行生成。这个问题也再次提醒后续每次扩展新领域时，要先补齐 IDL 再进入代码生成。

### 2. Windows 终端中的 MySQL 环境变量仍然存在会话刷新问题

在继续开发 category/tag 过程中，新的 PowerShell 会话仍然可能拿不到 `MYSQL_PASSWORD`，导致 `using password: NO`。当前阶段通过在当前终端临时补环境变量继续推进，后续可以考虑引入更稳定的配置读取方案。

### 3. 旧 proto 路线遗留文件容易干扰当前结构认知

随着项目正式切换到 `thrift`，旧的 `proto` 文件和迁移过渡期间留下的 `pb.go.bak` 文件继续保留在仓库里已经没有价值，反而容易让目录结构显得混乱，因此本阶段将它们移除。

## 当前结果

当前项目已经具备以下能力：

- 文章模块数据库化运行
- 分类模块可独立创建和查询
- 标签模块可独立创建和查询
- 文章创建与更新会校验分类是否真实存在
- 数据结构已经从“单文章字段”逐步演进到“文章 + 分类 + 标签”的关系化方向

## 下一步

下一阶段建议优先推进以下方向：

1. 将文章中的 `tag_ids` 从字符串字段切换到 `article_tags` 关联表
2. 在文章创建、更新、查询过程中引入真实标签关系读写
3. 让文章详情和列表能够返回更完整的分类/标签信息
4. 开始进入上传模块，为封面图和文章资源提供本地存储支持
5. 评估是否需要引入更稳定的配置管理方式，减少环境变量刷新问题
