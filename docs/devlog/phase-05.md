# Phase 05

## 目标

为管理端建立 `RBAC` 最小闭环方案，并把最近一轮统一错误返回的基础能力纳入本阶段整理，确保后续后台接口继续扩展时不会反复返工。

## 完成内容

1. 为接口错误处理补充结构化业务错误模型
2. 新增统一的成功/失败响应写出方法
3. 优化 JWT 中间件的未登录与无效 token 响应文案
4. 为文章创建、更新补充更明确的错误映射
5. 为分类、标签创建补充重复数据冲突映射
6. 起草 `RBAC` 最小闭环方案文档
7. 起草 `rbac.thrift` 第一版接口草案
8. 将 `RBAC` 领域补充进项目规范

## 设计决策

### 1. 先做最小 RBAC，而不是一步到位做后台系统

当前项目真正需要的是“后台接口权限开始变多时，系统仍然稳定可控”，因此这一阶段优先建立：

- 用户与角色关系
- 角色与权限关系
- 路由权限拦截

而不是先做复杂的角色页面或完整后台管理系统。

### 2. auth 和 rbac 分开

`auth` 继续负责：

- 登录
- JWT 签发
- JWT 解析

`rbac` 负责：

- 角色
- 权限点
- 绑定关系
- 权限校验

这样后续如果演进到微服务，也更容易拆分。

### 3. 错误体验先统一，再做更多权限逻辑

如果没有统一错误返回，后续引入 `401`、`403`、权限不足、资源冲突等场景时，前端和 Swagger 体验都会很混乱。因此先把错误结构收口，再进入权限模块更合理。

## 遇到的问题

### 1. Windows 沙箱命令执行存在不稳定情况

本阶段在读取和检查本地文件时，默认沙箱执行器出现了本地 SID 解析问题，因此改为在必要时使用提升方式完成检查和构建验证。问题不影响项目代码本身。

### 2. 旧代码仍然依赖兼容式 response/errno 写法

为了避免这一阶段改动过大，当前采取“兼容式演进”：

- 保留旧响应结构
- 增加新的结构化错误能力
- 先让关键接口切换到统一写法

这样不会因为一次性大改导致已有接口全面波动。

## 当前结果

当前项目已经额外具备：

- 更清晰的 `400 / 401 / 404 / 409 / 500` 业务错误表达
- 更明确的 JWT 失败原因提示
- 一份可直接进入实现阶段的 `RBAC` 规划文档
- 一份与当前项目风格一致的 `rbac.thrift` 草案
